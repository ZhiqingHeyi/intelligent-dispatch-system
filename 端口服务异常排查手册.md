## 1. 案例回顾：8123端口服务异常排查

### 1.1 问题现象

- 用户反馈端口8123的服务（混凝土车辆智慧运输系统）出现异常
- 该服务是生产环境中的关键业务系统

### 1.2 服务架构梳理

```
用户浏览器
    │
    ▼
┌─────────────────────────────────────┐
│  Nginx (8123端口)                    │
│  前端静态文件服务                      │
│  /data/project/qtb/datum/web/dist   │
└─────────────────────────────────────┘
    │
    ├──▶ 静态页面请求 (直接返回)
    │
    └──▶ /api/ 请求 (反向代理)
            │
            ▼
    ┌─────────────────────────────────────┐
│  Nginx (8125端口) - API网关           │
└─────────────────────────────────────┘
            │
            ▼
    ┌─────────────────────────────────────┐
│  Node.js (3000端口)                   │
│  混凝土温度监控系统后端                │
│  /home/python_env/temp/server         │
└─────────────────────────────────────┘
```

**关键发现**：8123 → 8125 → 3000 三层代理架构

## 2. 排查流程

### 2.1 第一步：确认问题范围

```bash
# 1. 检查端口是否监听
ss -tlnp | grep 8123

# 预期输出（正常）
LISTEN  0  128  *:8123  *:*  users:(("nginx",pid=xxx,fd=x))

# 2. 检查端口响应
curl -s -o /dev/null -w '%{http_code}' http://127.0.0.1:8123/
# 应该返回 200
```

**排查要点**：
- 如果端口未监听 → 服务未启动
- 如果返回非200 → 进一步检查后端

### 2.2 第二步：分层定位问题

#### 2.2.1 检查Nginx层（8123/8125）

```bash
# 检查Nginx配置
cat /usr/local/nginx/conf/nginx.conf | grep -A 20 'listen.*8123'

# 检查Nginx进程
ps aux | grep nginx

# 测试Nginx配置语法
/usr/local/nginx/sbin/nginx -t
```

**常见问题**：
- Nginx配置语法错误
- upstream后端不可达
- 权限问题

#### 2.2.2 检查后端服务层（3000端口）

```bash
# 检查3000端口监听
ss -tlnp | grep 3000

# 检查Node进程
ps aux | grep 'node.*app.js'

# 直接测试Node服务
curl -s http://127.0.0.1:3000/
```

**关键发现（本次案例）**：
```
LISTEN  0  128  [::]:3000  [::]:*  users:(("node",pid=44180,fd=19))
```

Node服务进程PID为44180，已运行多日。

### 2.3 第三步：查看服务日志

```bash
# Node.js服务日志（如果有配置）
tail -100 /home/python_env/temp/server/app.log

# 系统日志
journalctl -xe | grep node

# Nginx错误日志
tail -100 /usr/local/nginx/logs/error.log
```

**本次案例发现**：
```
Error: listen EADDRINUSE: address already in use :::3000
```

这表明Node服务尝试重启时端口被占用，说明服务可能曾经异常退出过。

### 2.4 第四步：服务重启验证

```bash
# 1. 找到进程ID
pgrep -f 'node.*app.js'
# 输出：44180

# 2. 查看进程启动时间
ps -eo pid,comm,lstart | grep 44180

# 3. 优雅停止服务
kill 44180

# 4. 等待进程退出
sleep 2

# 5. 验证端口释放
ss -tlnp | grep 3000
# 应该无输出

# 6. 重新启动服务
cd /home/python_env/temp/server
nohup node src/app.js > app.log 2>&1 &

# 7. 验证新进程
ps aux | grep 'node.*app.js'
```

**重启后验证**：
```bash
# 测试8123完整链路
curl -s http://127.0.0.1:8123/api/status | head -5

# 测试3000直接访问
curl -s http://127.0.0.1:3000/
```

## 3. 根本原因分析

### 3.1 本次案例结论

| 排查项 | 结果 | 结论 |
|--------|------|------|
| Python包降级影响 | ❌ 无关 | Node.js不依赖Python |
| Nginx配置修改 | ❌ 无关 | 配置修改时间为2025年7月 |
| Node服务运行时长 | ⚠️ 过长 | 从4月23日运行至今 |
| 系统资源耗尽 | ❌ 否 | 内存/CPU正常 |

**最终原因**：Node.js服务长时间运行（约4天），可能出现了内存泄漏或资源耗尽，需要定期重启。

### 3.2 类似问题常见原因

1. **内存泄漏**
   - 长时间运行导致内存占用持续增长
   - 最终触发OOM killer或响应变慢

2. **连接池耗尽**
   - 数据库连接未正确释放
   - 文件句柄达到上限

3. **依赖服务故障**
   - MySQL数据库连接中断
   - Redis缓存服务异常

4. **日志文件过大**
   - 磁盘空间不足
   - 日志写入失败导致服务阻塞

## 4. 排查命令速查表

### 4.1 端口和进程检查

```bash
# 查看端口监听
ss -tlnp | grep <端口号>

# 查看端口被哪个进程占用
lsof -i :<端口号>

# 查看进程详细信息
ps -ef | grep <进程名>
ps -eo pid,ppid,cmd,%mem,%cpu --sort=-%mem | head

# 查看进程启动时间
ps -eo pid,comm,lstart | grep <进程名>

# 查看进程打开的文件数
ls -l /proc/<PID>/fd | wc -l
```

### 4.2 服务状态检查

```bash
# 测试HTTP响应
curl -s -o /dev/null -w '%{http_code}' http://127.0.0.1:<端口>/

# 查看响应时间
curl -s -o /dev/null -w '%{time_total}' http://127.0.0.1:<端口>/

# 查看详细响应信息
curl -v http://127.0.0.1:<端口>/ 2>&1 | head -20

# 测试特定API接口
curl -s http://127.0.0.1:<端口>/api/status | head -100
```

### 4.3 日志查看

```bash
# 实时查看日志
tail -f /path/to/app.log

# 查看最近100行
tail -100 /path/to/app.log

# 查看包含错误关键字的日志
grep -i error /path/to/app.log | tail -20

# 查看系统日志
journalctl -xe | grep <服务名>

# 查看指定时间范围的日志
journalctl --since "1 hour ago" -u <服务名>
```

### 4.4 系统资源检查

```bash
# 内存使用
free -h

# CPU和内存占用最高的进程
top -bn1 | head -20

# 磁盘空间
df -h

# 磁盘Inode使用情况
df -i

# 查看文件句柄限制
ulimit -n
cat /proc/sys/fs/file-max

# 查看当前打开的文件总数
lsof | wc -l
```

## 5. 预防措施

### 5.1 配置服务自动重启

#### Node.js服务systemd配置示例

```ini
# /etc/systemd/system/temperature-monitor.service
[Unit]
Description=混凝土温度监控系统
After=network.target

[Service]
Type=simple
User=root
WorkingDirectory=/home/python_env/temp/server
ExecStart=/usr/bin/node src/app.js
Restart=always
RestartSec=10
StandardOutput=journal
StandardError=journal

# 内存限制（防止内存泄漏拖垮系统）
MemoryLimit=1G

[Install]
WantedBy=multi-user.target
```

**启用服务**：
```bash
systemctl daemon-reload
systemctl enable temperature-monitor
systemctl start temperature-monitor
```

### 5.2 配置定时重启

```bash
# 编辑crontab
crontab -e

# 每天凌晨3点重启服务（避免业务高峰期）
0 3 * * * systemctl restart temperature-monitor

# 或者每周一凌晨重启
0 3 * * 1 systemctl restart temperature-monitor
```

### 5.3 配置健康检查

```bash
#!/bin/bash
# health_check.sh

URL="http://127.0.0.1:8123/api/status"
TIMEOUT=10

# 测试服务响应
if ! curl -s --max-time $TIMEOUT "$URL" > /dev/null; then
    echo "$(date): 服务异常，执行重启" >> /var/log/health_check.log
    systemctl restart temperature-monitor
fi
```

**配置定时健康检查**：
```bash
# 每5分钟检查一次
*/5 * * * * /path/to/health_check.sh
```

### 5.4 日志轮转配置

```bash
# 安装logrotate（如未安装）
yum install logrotate -y

# 配置文件 /etc/logrotate.d/temperature-monitor
/home/python_env/temp/server/*.log {
    daily
    rotate 7
    compress
    delaycompress
    missingok
    notifempty
    create 644 root root
    postrotate
        systemctl reload temperature-monitor
    endscript
}
```

## 7. 故障处理决策树

```
用户反馈服务异常
    │
    ▼
端口是否监听？ ──否──▶ 服务未启动，启动服务
    │是
    ▼
本地测试是否200？ ──否──▶ 检查服务日志
    │是
    ▼
是否间歇性异常？ ──是──▶ 检查资源使用（内存/CPU/连接数）
    │否
    ▼
仅特定接口异常？ ──是──▶ 检查后端依赖（数据库/缓存）
    │否
    ▼
检查Nginx错误日志 ──发现错误──▶ 修复Nginx配置
    │
    无错误
    ▼
检查后端服务日志 ──发现错误──▶ 根据错误类型处理
    │
    无错误
    ▼
重启服务验证 ──解决问题──▶ 可能是内存泄漏，配置自动重启
    │
    未解决
    ▼
检查系统资源 ──资源不足──▶ 扩容或优化
    │
    正常
    ▼
联系开发人员进一步排查
```

## 8. 经验教训总结

### 8.1 本次案例收获

1. **不要急于归因**
   - 8123服务异常最初怀疑是Python包降级导致
   - 实际排查发现Node.js与Python无关

2. **分层排查的重要性**
   - 8123（Nginx）→ 8125（Nginx）→ 3000（Node.js）
   - 每层都要独立验证

3. **查看服务运行时间**
   - 长时间运行的服务容易出现内存泄漏
   - 建议定期重启或使用systemd自动重启

4. **日志是最好的朋友**
   - 应用日志、系统日志、Nginx日志都要查看
   - `EADDRINUSE` 错误提示了服务重启历史

### 8.2 后续改进建议

1. 为关键业务服务配置systemd管理
2. 设置监控告警（端口不通、响应超时）
3. 配置日志轮转防止磁盘占满
4. 建立服务健康检查机制
5. 定期执行服务重启（低峰期）
