# 车辆状态识别与特征工程专项说明

## 一、概述

本文档描述智能卸料动态匹配系统中车辆状态识别的特征工程机制、双队列匹配机制，包括当前实现方案、关键参数、已知限制和后续优化方向。

---

## 二、车辆状态定义

| 状态代码 | 中文标签 | 含义 | 触发条件 |
|---------|---------|------|---------|
| `loading` | 接料区装料 | 车辆在右端接料区等待/装料 | 位置在接料区 + 速度低 |
| `delivering` | 送料途中 | 车辆正从右端向左端送料 | Y坐标趋势减小 + 在途中或接料区出发 |
| `unloading` | 卸料区卸料 | 车辆在左端卸料区卸料 | 位置在卸料区 + 速度低 |
| `returning` | 返程途中 | 车辆正从左端向右端返程 | Y坐标趋势增大 + 在途中或卸料区出发 |
| `stopped` | 停止 | 车辆静止 | 速度为0且无明确方向趋势 |

## 三、坐标系与位置区域

### 3.1 车辆坐标系

车辆在Y字形分叉路上行驶，坐标系基于 `datum_data` 表的 `result_y` 字段：

```
右端(接料区)              途中区域              左端(卸料区)
   │                       │                       │
result_y ≈ -338      result_y -800~-400      result_y ≈ -1000
   │                       │                       │
   │   ← 送料(Y减小)       │                       │
   │   → 返程(Y增大)       │                       │
```

### 3.2 位置区域定义

| 区域 | result_y 范围 | 常量名 | 说明 |
|-----|--------------|--------|------|
| 接料区(右端) | (-400, 0) | `VEHICLE_LOADING_ZONE` | 车辆装料等待区 |
| 途中区域 | (-800, -400) | `VEHICLE_TRANSIT_ZONE` | 运输途中 |
| 卸料区(左端) | (-2000, -800) | `VEHICLE_UNLOADING_ZONE` | 卸料点区域 |

> ⚠️ **待验证**：以上区域边界值基于分析脚本估算，需在实际运行中观察数据分布进行校准。

## 四、方向识别算法

### 4.1 滑动窗口趋势法（当前实现）

**核心思路**：不依赖单帧差分，而是使用最近5帧的Y坐标累计变化量（趋势）判断方向。

```python
# 滑动窗口大小
VEHICLE_Y_HISTORY_SIZE = 5

# 计算趋势
y_trend = Σ(result_y[i] - result_y[i-1])  # i从1到窗口大小

# 判断方向
if y_trend < VEHICLE_GOING_THRESHOLD:      # -10
    direction = 'going'    # 送料
elif y_trend > abs(VEHICLE_GOING_THRESHOLD):
    direction = 'returning' # 返程
```

**优势**：
- 过滤单帧信号跳跃
- 反映持续运动趋势
- 窗口越大越稳定，但响应越慢

### 4.2 方向确认机制

防止信号短暂异常导致方向频繁切换：

```python
# 确认帧数
VEHICLE_DIRECTION_CONFIRM_FRAMES = 2

# 逻辑：新方向需连续出现2帧才确认切换
# 否则保持原方向
```

**效果**：车辆方向从 `going` 切换到 `returning` 需要连续2帧检测到返程趋势，避免误触发。

### 4.3 关键阈值参数

| 参数 | 当前值 | 常量名 | 说明 | 调优建议 |
|-----|-------|--------|------|---------|
| Y变化阈值 | -10 | `VEHICLE_GOING_THRESHOLD` | Y坐标累计变化超过此值才认为有方向 | 观察实际数据波动范围调整 |
| 确认帧数 | 2 | `VEHICLE_DIRECTION_CONFIRM_FRAMES` | 方向切换需连续确认的帧数 | 增大更稳定但响应慢 |
| 滑动窗口 | 5 | `VEHICLE_Y_HISTORY_SIZE` | Y坐标历史记录长度 | 增大更平滑但延迟高 |
| 速度阈值 | 0.5 | `VEHICLE_SPEED_THRESHOLD` | 判断车辆是否在移动 | 根据停车场景调整 |
| Y趋势阈值 | 10 | `VEHICLE_Y_DELTA_THRESHOLD` | 状态检测中Y趋势的最小有效变化 | 与GOING_THRESHOLD保持一致 |

## 五、任务完成OR逻辑

### 5.1 触发条件

任务完成判定采用 **OR逻辑**，以下任一条件满足即完成任务：

| 触发源 | 条件 | 含义 | 实现位置 |
|-------|------|------|---------|
| 触发源1 | 缆机 `loading → delivering` | 缆机装好料开始送料，说明车已到 | `sync_cable_cars()` |
| 触发源2 | 车辆 `going → returning` | 车辆送完料开始返程 | `sync_vehicles()` |

### 5.2 兜底机制

`auto_complete()` 作为兜底，每个同步周期检查：
- 车辆当前 `direction = 'returning'` 的 assigned 任务
- 缆机当前 `state = 'delivering'` 的 assigned 任务

防止状态切换事件丢失导致任务永远无法完成。

### 5.3 日志标识

| 日志标签 | 含义 |
|---------|------|
| `[COMPLETE-CABLE]` | 触发源1实时完成（缆机开始送料） |
| `[COMPLETE-VEHICLE]` | 触发源2实时完成（车辆开始返程） |
| `[COMPLETE-FALLBACK-CABLE]` | 兜底完成（缆机正在送料） |
| `[COMPLETE-FALLBACK-VEHICLE]` | 兜底完成（车辆已返程） |

## 七、已知限制与待验证项

### 7.1 位置区域边界

当前区域边界值（-400, -800）基于分析脚本估算，可能需要根据实际运行数据调整。

**验证方法**：
1. 观察日志中 `[VEHICLE-STATE]` 输出的方向切换
2. 对比前端显示的车辆状态与实际对讲机确认的情况
3. 统计各区域车辆分布，校准边界值

### 7.2 信号跳跃

车辆定位基站较少，`result_y` 可能出现跳跃，导致：
- 短暂的方向误判（已通过确认机制缓解）
- 趋势计算偏差（已通过滑动窗口缓解）

**后续优化方向**：
- 增加异常值过滤（如单帧变化超过100视为跳跃，剔除）
- 引入加权滑动窗口（近期数据权重更高）

### 7.3 Y字形分叉路

当前未区分上分支和下分支，仅依赖 `result_y` 的变化趋势。如果需要区分不同路线，可以：
- 利用 `result_x` 辅助判断分支
- 增加 `result_y` 的正负区间细分

### 7.4 车辆速度特征

当前未利用车辆速度差异区分空载/满载。后续可探索：
- 满载送料时速度较慢
- 空载返程时速度较快
- 结合速度模式辅助状态判断

## 十、双队列匹配机制

### 10.1 问题背景

原有匹配逻辑存在两个核心问题：

**问题A - 车先到排队等缆机**：
车辆已到达卸料平台等待，但缆机还没回来。缆机回来后，系统可能匹配到还在路上的车，而非已在平台等待的车。

**问题B - 缆机先到排队等车**：
多台缆机返回装料区等待，后到的缆机可能先匹配到车辆，先到的缆机反而继续等待。

### 10.2 队列设计

#### 缆机等待队列（cable_car_queue）

| 操作 | 条件 | 说明 |
|------|------|------|
| 入队 | 缆机 `returning → loading` | 到达装料区等待接料 |
| 出队 | 匹配成功 | 已找到车辆 |
| 出队 | `loading → delivering` | 已接完料出发（可能人工调度） |
| 出队 | 状态变为非loading | 异常清理 |

#### 车辆等待队列（vehicle_queue）

| 操作 | 条件 | 说明 |
|------|------|------|
| 入队 | `direction='going'` 且 `grade_id>0` 且未分配 | 送料中的车辆进入匹配池 |
| 优先级升级 | `state='unloading'` 或在平台低速等待 | 已到平台的车优先匹配 |
| 出队 | 匹配成功 | 已找到缆机 |
| 出队 | `direction='returning'` | 已卸完料返程 |
| 出队 | `grade_id=0` | 级配清零 |
| 出队 | 被分配 | 已被其他方式分配 |

### 10.3 车辆优先级

| 优先级 | 值 | 含义 | 判定条件 |
|-------|---|------|---------|
| `at_platform` | 0（最高） | 车辆已在平台等待 | `state='unloading'` 或在接料区低速等待 |
| `on_the_way` | 1 | 车辆还在路上 | `direction='going'` 但未到平台 |

### 10.4 匹配算法

```
1. 清理队列中的无效条目（已分配/已返程/级配变更等）
2. 缆机队列按入队时间排序（FIFO，先到先服务）
3. 车辆队列按 (优先级, 入队时间) 排序
4. 对每台缆机（FIFO顺序）：
   a. 从数据库读取最新级配
   b. 在车辆队列中找级配匹配的车辆
   c. 优先匹配 at_platform 的车，其次 on_the_way
   d. 同优先级内按入队时间FIFO
   e. 匹配成功：双方出队，创建任务
5. 未匹配的缆机继续等待
```

### 10.5 队列清理机制

每个匹配周期执行 `_cleanup_queues()`，确保队列与数据库状态一致：
- 移除已分配的缆机/车辆
- 移除已返程的车辆
- 移除级配清零的车辆
- 移除状态非loading的缆机
- 升级到达平台的车辆优先级
- 同步最新的grade_id

### 10.6 队列日志标识

| 日志标签 | 含义 |
|---------|------|
| `[QUEUE-CABLE-IN]` | 缆机入队 |
| `[QUEUE-CABLE-OUT]` | 缆机出队 |
| `[QUEUE-CABLE-UPDATE]` | 缆机级配更新 |
| `[QUEUE-VEHICLE-IN]` | 车辆入队 |
| `[QUEUE-VEHICLE-OUT]` | 车辆出队 |
| `[QUEUE-VEHICLE-UPGRADE]` | 车辆优先级升级 |
| `[QUEUE-CLEANUP]` | 队列清理 |
| `[MATCH-FIFO]` | FIFO匹配成功（含等待时长） |
| `[MATCH-WAITING]` | 缆机等待同级配车辆 |

### 10.7 匹配场景示例

**场景1：车先到，缆机后到**
```
T1: 车辆A(二级配) 到达平台 → 入队(priority=at_platform)
T2: 车辆B(二级配) 还在路上 → 入队(priority=on_the_way)
T3: 缆机1(二级配) 返回装料区 → 入队
T4: 匹配：缆机1 ↔ 车辆A（at_platform优先于on_the_way）✓
```

**场景2：缆机先到，车后到**
```
T1: 缆机1(二级配) 返回装料区 → 入队
T2: 缆机2(三级配) 返回装料区 → 入队
T3: 车辆A(三级配) 开始送料 → 入队
T4: 匹配：缆机1(二级配)无匹配车等待中，缆机2 ↔ 车辆A（级配匹配）✓
T5: 车辆B(二级配) 开始送料 → 入队
T6: 匹配：缆机1 ↔ 车辆B ✓
```

**场景3：跨级配不阻塞**
```
T1: 缆机1(二级配) 先到 → 队列位置1
T2: 缆机2(三级配) 后到 → 队列位置2
T3: 车辆A(三级配) 到达 → 入队
T4: 匹配：缆机1无二级配车跳过，缆机2 ↔ 车辆A ✓（不阻塞）
```

## 十二、vehicle_system.data 表数据写入

### 12.1 功能说明

智能匹配成功后，系统自动将匹配结果写入 `vehicle_system.data` 表的 `unloading_port` 字段，实现与第三方系统的数据互通。

### 12.2 表结构

| 字段 | 类型 | 说明 |
|------|------|------|
| `tid` | INT | 车辆ID（与 datum_data.tid 对应） |
| `receiving_port` | INT | 接料端口（已有字段） |
| `unloading_port` | INT | 分配的缆机ID（**本系统写入**） |

### 12.3 写入时机

| 操作 | 触发条件 | SQL操作 |
|------|----------|---------|
| **写入** | 匹配成功时 | `UPDATE data SET unloading_port = ? WHERE tid = ?` |
| **清空** | 任务完成时 | `UPDATE data SET unloading_port = NULL WHERE tid = ?` |

### 12.4 清空触发源

任务完成时（OR逻辑），自动清空 `unloading_port`：

1. **缆机开始送料** (`loading → delivering`)
2. **车辆开始返程** (`going → returning`)
3. **兜底机制** (`auto_complete` 定期检查)

### 12.5 日志标识

| 日志标签 | 含义 |
|---------|------|
| `[DB-WRITE]` | 成功写入/清空 unloading_port |
| `[DB-WRITE-WARN]` | 写入警告（如车辆不存在） |
| `[DB-WRITE-ERROR]` | 写入失败（异常） |

### 12.6 幂等性设计

- **重复匹配**：同一车辆被重复匹配时，会更新为最新的缆机ID（覆盖写入）
- **失败处理**：写入失败不影响主流程，仅记录错误日志
- **事务隔离**：使用独立的数据库连接，不影响本地SQLite事务

### 12.7 异常处理

```
1. 车辆tid在data表中不存在 → 记录警告日志，继续执行
2. 数据库连接失败 → 记录错误日志，不影响匹配流程
3. 写入超时 → 记录错误日志，不影响匹配流程
```
