# 缆机实时数据库读取机制技术文档

## 一、系统架构概述

本系统采用**双数据库架构**实现缆机实时数据的读取与处理：

```
┌─────────────────┐         ┌──────────────────┐         ┌─────────────────┐
│  外部MySQL数据库  │  ────→  │  数据同步引擎     │  ────→  │  本地SQLite数据库 │
│  (实时数据源)    │  定时读取  │  (data_sync.py)  │  写入    │  (dispatch.db)  │
└─────────────────┘         └──────────────────┘         └─────────────────┘
        ↑                                                           ↓
   缆机/车辆GPS数据                                            Web API接口
   实时更新                                                   (/api/status)
```

### 2.2 同步循环机制

**代码位置**: [data_sync.py#L708-L723](file:///c:/Users/1/Desktop/智能建造实验室/智能卸料动态匹配/dispatch_center/data_sync.py#L708-L723)

```python
def start_sync_loop():
    global _sync_thread, _sync_running
    if _sync_running:
        return
    _sync_running = True

    def _loop():
        while _sync_running:
            try:
                sync_all()  # 执行所有同步操作
            except Exception as e:
                print(f"[SYNC] loop error: {e}")
            time.sleep(SYNC_INTERVAL)  # SYNC_INTERVAL = 3秒

    _sync_thread = threading.Thread(target=_loop, daemon=True)
    _sync_thread.start()
```

**特点**：
- 使用独立线程运行，不阻塞主线程
- 每**3秒**执行一次完整同步（`SYNC_INTERVAL = 3`）
- 异常捕获确保循环不会因单次错误而中断
- 守护线程（daemon=True），随主程序退出而终止

### 4.3 车辆方向检测算法

**代码位置**: [data_sync.py#L80-L146](file:///c:/Users/1/Desktop/智能建造实验室/智能卸料动态匹配/dispatch_center/data_sync.py#L80-L146)

#### 核心原理

通过维护**Y坐标滑动窗口**（最近5个数据点），计算Y坐标的变化趋势来判断行驶方向：

```python
VEHICLE_Y_HISTORY_SIZE = 5          # 历史窗口大小
VEHICLE_DIRECTION_CONFIRM_FRAMES = 2  # 方向确认帧数
VEHICLE_GOING_THRESHOLD = -10       # 去程阈值
```

#### 判断逻辑

```
Y坐标趋势计算：
history = [y₁, y₂, y₃, y₄, y₅]  （最新的5个Y值）
y_trend = (y₂-y₁) + (y₃-y₂) + (y₄-y₃) + (y₅-y₄)

如果 y_trend < -10  →  going（去程，Y值减小）
如果 y_trend > 10   →  returning（返程，Y值增大）
其他情况             →  idle（静止或不确定）
```

**防抖动机制**：
- 方向改变需要连续**2帧**确认才生效
- 使用 `_vehicle_direction_confirm` 字典跟踪确认计数
- 车辆停止时保留最后已知方向（配合状态锁定）

### 4.4 车辆状态识别（带状态锁定）

**代码位置**: [state_detector.py#L105-L209](file:///c:/Users/1/Desktop/智能建造实验室/智能卸料动态匹配/dispatch_center/state_detector.py#L105-L209)

#### 设计背景

GPS信号在施工现场可能不稳定，导致坐标跳变。为避免状态频繁跳变，引入**状态锁定机制**：

> **核心理念**：状态一旦确认就"锁定"，只有满足明确的转换条件才能变更。

#### Y坐标位置区域定义

```
result_y（Y坐标）轴：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━→
-1400        -950        50         120
│←──卸料区──→│←────送料途中────→│←─接料区─→│
(unloading)  (delivering)      (loading)
```

**位置区域划分**:
- **接料区** (loading_zone): Y ∈ [50, 120]
- **送料途中** (delivering_zone): Y ∈ [-950, 50]
- **卸料区** (unloading_zone): Y ∈ [-1400, -950]

#### 状态流转图

```
                    ┌─────────────┐
                    │   standby   │ ← 待命（在接料区）
                    └──────┬──────┘
                           │ 条件：离开接料区(Y<50) + 方向going
                           ▼
                    ┌─────────────┐
                    │  delivering │ ← 送料中
                    └──────┬──────┘
                           │ 条件：到达卸料区(Y<-950)
                           ▼
              ┌────────────┴────────────┐
              ▼                         ▼
     ┌─────────────┐           ┌─────────────────┐
     │  unloading  │           │ delivering_pause│ ← 送料暂停
     │  (卸料中)    │           └────────┬────────┘
     └──────┬──────┘                    │ 条件：恢复移动+方向going
            │                           └──────────────┘
            │ 条件：停止卸料
            ▼
     ┌─────────────────┐
     │ unloading_wait  │ ← 卸料区等待
     └────────┬────────┘
              │ 条件：离开卸料区 + 方向returning
              ▼
     ┌─────────────┐
     │  returning  │ ← 返程中
     └──────┬──────┘
            │ 条件：回到接料区(Y>50)
            └──────→ 回到 standby
```

#### 锁定机制工作原理

```python
# 全局状态存储
_vehicle_locked_state = {}           # {tid: locked_state}
_vehicle_state_confirm_count = {}    # {tid_old_new: count}
STATE_CONFIRM_THRESHOLD = 3          # 需要连续3帧确认
```

**处理流程**：

1. **计算原始状态（raw_state）**：基于当前帧的位置和方向
2. **获取锁定状态（locked_state）**：查看该车辆当前锁定的状态
3. **比较两者**：
   - 如果相同 → 直接返回锁定状态
   - 如果不同 → 检查是否允许转换（`_can_transition()`）
4. **状态转换验证**：
   - 不允许转换 → 保持锁定状态（忽略GPS波动）
   - 允许转换 → 需要**连续3帧**确认才真正转换
5. **返回结果**：始终返回锁定状态（可能是原始状态或保持旧状态）

**示例场景**：
```
帧1: GPS显示Y=-200, 方向going → raw_state=delivering, locked=standby
     → 不允许 standby→delivering（未离开接料区）→ 返回 standby ✓

帧2: GPS显示Y=60（跳回接料区！） → raw_state=standby, locked=delivering
     → 不允许 delivering→standby（逆向转换不允许）→ 返回 delivering ✓
     → ✅ 成功过滤GPS异常！

帧3-5: GPS稳定在Y=-1000（卸料区） → raw_state=unloading_wait
     → 允许 delivering→unloading_wait
     → 帧3: confirm_count=1/3
     → 帧4: confirm_count=2/3
     → 帧5: confirm_count=3/3 ✓ 达到阈值！
     → 锁定状态转换为 unloading_wait
```

### 6.2 自动匹配引擎

**代码位置**: [data_sync.py#L538-L627](file:///c:/Users/1/Desktop/智能建造实验室/智能卸料动态匹配/dispatch_center/data_sync.py#L538-L627)

每次数据同步成功后，系统会自动执行匹配算法：

```python
def auto_match():
    """自动匹配逻辑 - FIFO + 优先级队列匹配"""
    # 1. 清理无效队列条目
    _cleanup_queues(db)

    # 2. 缆机队列按入队时间排序（FIFO）
    _cable_car_queue.sort(key=lambda x: x['enter_time'])

    # 3. 车辆队列按优先级排序（at_platform优先 → 入队时间）
    _vehicle_queue.sort(key=lambda x: (x['priority'], x['enter_time']))

    # 4. 为每台缆机寻找同级配的车辆
    for car_item in _cable_car_queue:
        for v_item in _vehicle_queue:
            if v_item['grade_id'] == car_item['grade_id']:
                # 匹配成功！创建任务
                create_dispatch_task(car_id, vehicle_id, grade_id)
                # 发送消息给车辆
                send_match_message_to_vehicle(...)
                # 更新车辆的unloading_port字段
                _update_vehicle_unloading_port(tid, car_id)
                break
```

**匹配队列管理**：
- **缆机入队条件**：状态从 returning → loading（到达装料区）
- **缆机出队条件**：匹配成功 / 开始送料 / 状态异常
- **车辆入队条件**：方向='going' 且 grade_id>0 且未分配
- **车辆出队条件**：匹配成功 / 返程 / 级配清零 / 已分配

## 十、总结

### 核心优势

1. **实时性强**：3秒同步周期，接近实时监控
2. **可靠性高**：状态锁定机制有效应对GPS信号波动
3. **智能化**：FIFO+优先级队列匹配算法，兼顾效率与公平
4. **容错性好**：完善的异常处理和队列清理机制
5. **可扩展**：模块化设计，易于添加新功能

### 技术亮点

- **双缓冲架构**：外部MySQL（数据源）+ 本地SQLite（业务层），解耦读写
- **状态机模式**：缆机和车辆都采用有限状态机建模
- **滑动窗口算法**：用于车辆方向检测，抗干扰能力强
- **确认机制**：状态/方向变更需要多帧确认，防止误判
- **OR逻辑完成任务**：多种触发源，提高任务完成的及时性

### 适用场景

本系统适用于：
- 水电工程缆机送料调度
- 施工现场车辆管理
- 实时定位监控系统
- 多资源匹配优化问题
