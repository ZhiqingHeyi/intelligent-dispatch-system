## 1. 系统概述

### 1.1 系统名称
**智能卸料动态匹配系统 (Intelligent Unloading Dynamic Matching System)**

### 1.2 功能描述
- 实时监控缆车和车辆位置状态

### 4.2 数据流说明

1. **前端 → 后端**: Vue通过Axios发送HTTP请求到Flask API
2. **后端 → 本地SQLite**: 任务数据、AI配置、聊天历史存储在本地
3. **后端 → 外部MySQL**: 
   - 缆车位置数据从 `cable_car` 数据库读取
   - 车辆位置数据从 `vehicle_system` 数据库读取
4. **后端 → OpenAI API**: AI调度功能调用外部AI模型

## 6. 服务配置详解

### 6.1 后端服务配置

**文件位置**: `/etc/systemd/system/dispatch-center-backend.service`

```ini
[Unit]
Description=智能卸料动态匹配系统后端服务  
After=network.target

[Service]
Type=simple
User=root
WorkingDirectory=/opt/dispatch-center/dispatch_center
Environment="PATH=/opt/dispatch-center/venv/bin"
ExecStart=/opt/dispatch-center/venv/bin/python /opt/dispatch-center/dispatch_center/app.py
Restart=always
RestartSec=10
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

**关键配置说明**:
- `Environment`: 使用虚拟环境的Python解释器，隔离系统依赖
- `Restart=always`: 崩溃后自动重启
- `RestartSec=10`: 重启间隔10秒
- 日志输出到systemd journal

### 6.2 前端服务配置

**文件位置**: `/etc/systemd/system/dispatch-center-frontend.service`

```ini
[Unit]
Description=智能卸料动态匹配系统前端服务  
After=network.target

[Service]
Type=simple
User=root
WorkingDirectory=/opt/dispatch-center/vue-frontend-dist
ExecStart=/usr/bin/python3 -m http.server 9023 --bind 0.0.0.0
Restart=always
RestartSec=10
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

**关键配置说明**:
- 使用Python内置的HTTP服务器托管静态文件
- 绑定所有网络接口 (`0.0.0.0`) 允许外部访问
- 无需虚拟环境（纯静态文件服务）

4. ✅ 安装Python依赖（Flask, requests, openai, pymysql）
5. ✅ 创建systemd服务配置
6. ✅ 启动并启用服务

#### 步骤3: 验证部署
```bash
# 检查服务状态
```

## 11. 常见问题排查

### 11.1 ❌ 端口被占用

**症状**:
```
OSError: [Errno 98] Address already in use
```

**排查步骤**:
```bash
# 1. 查看哪个进程占用了端口
ss -tlnp | grep :9045
# 输出: LISTEN 0 128 *:9045 users:(("python",pid=12345,fd=3))

# 2. 杀掉占用进程
kill -9 12345

# 3. 或找到对应的服务停止
systemctl stop dispatch-center-backend

# 4. 重新启动
systemctl start dispatch-center-backend
```

### 11.2 ❌ 服务启动失败

**症状**:
```
Failed to start dispatch-center-backend.service
```

**排查步骤**:
```bash
# 1. 查看详细错误日志
journalctl -u dispatch-center-backend -n 50 --no-pager

# 2. 手动运行查看报错
cd /opt/dispatch-center/dispatch_center
/opt/dispatch-center/venv/bin/python app.py

# 3. 常见原因：
#    - 语法错误（检查Python文件是否有语法错误）
#    - 缺少依赖（pip install缺失的包）
#    - 端口被占用（见11.1）
#    - 数据库权限问题（chmod 644 dispatch.db）
```

### 11.3 ❌ API返回404 NOT FOUND

**症状**:
```
POST http://192.168.1.88:9045/api/task/872/complete - 404 (NOT FOUND)
```

**原因分析**:
- 前端调用的API路径在后端不存在
- HTTP方法不匹配（GET vs POST vs PUT）

**解决方案**:
1. 检查前端API调用代码 (`vue-frontend/src/api/dispatch.ts`)
2. 检查后端路由定义 (`dispatch_center/app.py`)
3. 确保路径和方法完全一致
4. 如需兼容，添加新的路由处理函数

**示例修复**:
```python
# app.py 中添加兼容路由
@app.route('/api/task/<int:task_id>/complete', methods=['POST'])
def complete_task_v2(task_id):
    complete_dispatch_task(task_id)
    return jsonify({'success': True})
```

### 11.4 ❌ API返回500 INTERNAL SERVER ERROR

**症状**:
```
POST http://192.168.1.88:9045/api/ai/scheduler - 500 (INTERNAL SERVER ERROR)
```

**排查步骤**:

**Step 1: 查看后端错误日志**
```bash
journalctl -u dispatch-center-backend -f --no-pager
# 寻找 Traceback 和 Error 信息
```

**Step 2: 常见500错误及修复**

#### 错误A: 字符串语法错误
```
SyntaxError: unterminated string literal (detected at line 7) (ai_experience.py, line 7)
```
**原因**: Python文件中有截断/损坏的字符串
**修复**: 检查并修正字符串字面量（如 `'二级�?'` → `'二级配'`）

#### 错误B: 数据库锁定
```
sqlite3.OperationalError: database is locked
```
**原因**: SQLite并发写入冲突
**修复**:
- 已启用WAL模式（Write-Ahead Logging）
- 增加超时时间：`timeout=10`
- 如果频繁出现，考虑降低SYNC_INTERVAL

#### 错误C: 函数名未定义
```
NameError: name '_get_db' is not defined
```
**原因**: 函数名拼写错误或作用域问题
**修复**: 检查函数定义和调用是否一致

**Step 3: 测试API**
```bash
# 直接测试API获取详细错误
curl -v http://192.168.1.88:9045/api/ai/scheduler
```

### 11.5 ❌ CORS跨域错误

**症状**:
```
Access-Control-Allow-Origin header is missing
```

**原因**: 前端（9023端口）访问后端（9045端口）被浏览器阻止

**解决方案**: 
已在 `app.py` 中添加CORS头：
```python
@app.after_request
def after_request(response):
    response.headers.add('Access-Control-Allow-Origin', '*')
    response.headers.add('Access-Control-Allow-Headers', 'Content-Type,Authorization,X-Requested-With')
    response.headers.add('Access-Control-Allow-Methods', 'GET,PUT,POST,DELETE,OPTIONS')
    return response
```

如仍有问题，检查：
1. 后端服务是否正常运行
2. 是否有Nginx/反向代理覆盖了响应头

### 11.6 ❌ 数据库连接问题

**症状**:
```
Cannot operate on a closed database
OperationalError: unable to open database file
```

**原因**: 
- 数据库连接在多线程中共享导致关闭冲突
- 数据库文件权限不正确
- 磁盘空间不足

**解决方案**:
1. **使用独立的数据库连接**（每个线程自己的连接）:
```python
def _get_db():
    conn = sqlite3.connect(DB_PATH, timeout=10)
    conn.row_factory = sqlite3.Row
    return conn
```

2. **修复文件权限**:
```bash
chown root:root /opt/dispatch-center/dispatch_center/dispatch.db
chmod 644 /opt/dispatch-center/dispatch_center/dispatch.db
```

3. **清理WAL文件**（如果数据库损坏）:
```bash
cd /opt/dispatch-center/dispatch_center
rm -f dispatch.db-wal dispatch.db-shm
systemctl restart dispatch-center-backend
```

### 14.2 🟡 网络安全

**当前风险**:
- 使用HTTP而非HTTPS（明文传输）
- 端口直接暴露（无防火墙规则限制）
- root用户运行服务

**建议加固**:
```bash
# 添加防火墙规则（仅允许内网访问）
firewall-cmd --permanent --add-rich-rule='rule family="ipv4" source address="192.168.1.0/24" port port="9023" protocol="tcp" accept'
firewall-cmd --permanent --add-rich-rule='rule family="ipv4" source address="192.168.1.0/24" port port="9045" protocol="tcp" accept'
firewall-cmd --reload

# 或者使用iptables
iptables -A INPUT -p tcp --dport 9023 -s 192.168.1.0/24 -j ACCEPT
iptables -A INPUT -p tcp --dport 9045 -s 192.168.1.0/24 -j ACCEPT
iptables -A INPUT -p tcp --dport 9023 -j DROP
iptables -A INPUT -p tcp --dport 9045 -j DROP
```

### 14.3 🟢 最小权限原则

**当前状态**: 使用root用户运行服务（开发阶段可接受）

**生产环境建议**:
```bash
# 创建专用用户
useradd -r -s /bin/false dispatchuser

# 修改服务配置
# WorkingDirectory保持不变
# User改为dispatchuser
```
