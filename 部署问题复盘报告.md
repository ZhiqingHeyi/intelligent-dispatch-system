## 2. 核心问题分析

### 2.1 Python包版本降级

#### 2.1.1 受影响的包

| 包名 | 原始版本 | 部署后版本 | 差异 |
|------|----------|------------|------|
| **requests** | 2.32.3 | 2.31.0 | 安全修复版本被回退 |
| **flask** | 3.0.3 | 3.0.0 | Bug修复版本被回退 |
| **pymysql** | 1.1.2 | 1.1.0 | 功能更新被回退 |

#### 2.1.2 根本原因

部署脚本 `deploy-no-nginx.sh` 中使用了**固定版本号**：

```bash
# 问题代码（requirements.txt）
cat > requirements.txt << 'EOF'
flask==3.0.0      # ❌ 固定版本，强制降级
requests==2.31.0   # ❌ 固定版本，强制降级
pymysql==1.1.0     # ❌ 固定版本，强制降级
openai==1.6.1
EOF
```

当执行 `pip3 install -r requirements.txt` 时，pip强制将系统中已有的高版本包降级为requirements.txt中指定的低版本。

#### 2.1.3 影响范围

- **直接影响**: 所有依赖这些Python包的服务可能受到影响
- **潜在风险**: 
  - 安全漏洞（requests 2.32.3包含安全修复）
  - 功能异常（flask 3.0.3包含bug修复）
  - 数据库兼容性问题

## 5. 改进建议

### 5.1 依赖管理优化

#### 方案一：放宽版本限制（推荐）

```bash
# 改进前 ❌
flask==3.0.0
requests==2.31.0
pymysql==1.1.0

# 改进后 ✅
flask>=3.0.0
requests>=2.31.0
pymysql>=1.1.0
```

#### 方案二：使用虚拟环境（最佳实践）

```bash
# 创建独立虚拟环境
python3 -m venv /opt/dispatch-center/venv

# 激活环境
source /opt/dispatch-center/venv/bin/activate

# 安装依赖（不影响系统Python）
pip install -r requirements.txt
```

#### 方案三：使用Conda环境

```bash
# 创建conda环境
conda create -n dispatch-center python=3.12

# 激活环境
conda activate dispatch-center

# 安装依赖
pip install -r requirements.txt
```

## 6. 责任判定

### 6.1 我们的责任

| 问题 | 责任 | 说明 |
|------|------|------|
| Python包降级 | ✅ 我们的责任 | 部署脚本使用了固定低版本号 |
| 8123服务异常 | ❌ 非我们的责任 | Node服务自身需要重启，与Python包无关 |

### 6.2 经验教训

1. **永远不要假设目标环境的Python包版本**
   - 部署前检查现有版本
   - 使用虚拟环境隔离依赖

2. **requirements.txt版本号策略**
   - 使用 `>=` 代替 `==`
   - 或完全不指定版本，让pip自动选择兼容版本

3. **生产环境部署原则**
   - 先在测试环境验证
   - 准备回滚脚本
   - 选择低峰期部署
